after a successful three-way handshake, the server often sends a banner for identification.
this serves to let the client know which service it is working with.

if we manually connect to the service port using netcat, grab the banner, and intercept the network traffic using tcpdump
with that, we can see what nmap would not show us when they can't identify the service by signature.

making the tcp handshake
```
sudo tcpdump -i eth0 host <client_ip> and <remote_server_ip>
```

intercepting network traffic to grab banner after SYN ACK
```
nc -nv <remote_server_ip> <port>
```

SMTP example:
```
root@root[/]$ tcpdump -i eth0 host 10.10.14.2 and 10.129.2.28

tcpdump: verbose output suppressed, use -v or -vv for full protocol decode
listening on eth0, link-type EN10MB (Ethernet), capture size 262144 bytes
```

```
root@root[/]$ nc -nv 10.129.2.28 25

Connection to 10.129.2.28 port 25 [tcp/*] succeeded!
220 inlane ESMTP Postfix (Ubuntu)
```
